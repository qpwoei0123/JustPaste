JustPaste

링크를 붙여넣고, 선택과 승인은 사람에게 남기고, 적용은 에이전트가 한다.

이 페이지는 사람이 읽어도 이해돼야 하고, 에이전트가 읽어도 apply 순서를 놓치지 않아야 한다.

핵심 원칙: 목록만 보고 바로 apply하지 말고, 선택 → detail/instruction 재조회 → scope 확인 → dry-run → 명시 승인 순서를 지킨다.

첫 적용 플로우

  1. ./api/v1/packs/index.json을 읽고 선택 가능한 pack 목록을 먼저 보여준다.
  2. 사용자가 pack을 고르면, 선택된 pack의 detail.json과 instruction.json을 다시 읽는다.
  3. 에이전트는 적용 범위와 예상 변경 파일, 위험도, 롤백 힌트를 요약한다.
  4. 사용자가 명시적으로 승인하면 그때 apply하고 결과를 보고한다.

사용자가 직접 결정하는 것

1. 무엇을 적용할지

pack 이름, id, summary, risk, 지원 scope를 먼저 본 뒤 선택한다.

2. 어디에 적용할지

project인지 global인지 먼저 정한다. 지원하지 않는 scope는 강제로 밀어넣지 않는다.

3. 덮어쓸지 병합할지

기존 파일이 있다면 overwrite/merge 여부를 재확인한다.

4. 지금 적용할지

dry-run 요약과 변경 파일 목록을 본 뒤 apply를 승인한다.

명시 승인 예시

애매한 답변은 승인으로 간주하지 않는다. 특히 첫 적용에서는 더 보수적으로 본다.

  • 좋은 예: yes, apply
  • 좋은 예: yes, overwrite and apply
  • 좋은 예: global scope로 적용해
  • 부족한 예: ㅇㅇ, 그거, ㄱㄱ, 진행

Published Packs

Commit Convention Skill
skill id: commit-convention

Purpose: Standardize commit messages with a conventional format for clearer history and collaboration.

  1. Check existing commit style in the current repository.
  2. Present base format: <type>(<scope>): <summary>.
  3. Ask user for apply scope (project/global).
  4. If rule files exist, ask overwrite confirmation.
  5. Apply and provide two sample commit messages.
PR Clarity Template Skill
skill id: pr-clarity-template

Purpose: Keep PR descriptions concise and review-friendly with a consistent structure.

  1. Check whether a PR template already exists.
  2. Present sections: Summary, Changes, Scope, Validation, Related Issue.
  3. Ask user for apply scope (project/global).
  4. Ask merge/replace choice if a template already exists.
  5. Apply and provide one sample PR body.
