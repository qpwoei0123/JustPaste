JustPaste

Paste the link, leave selection and approval to the human, and let the agent apply the setup.

This page should be understandable to humans and should keep agents from skipping the apply order.

Core rule: do not apply from the pack list alone. Follow selection → refetch detail/instruction → confirm scope → dry-run → explicit approval.

First Apply Flow

  1. Read ./api/v1/packs/index.json and show the list of available packs first.
  2. After the user chooses a pack, read that pack's detail.json and instruction.json again.
  3. The agent summarizes the apply scope, expected changed files, risk, and rollback hints.
  4. Only after explicit approval does the agent apply and report the result.

User Decisions

1. What to apply

Choose after reviewing the pack name, id, summary, risk, and supported scopes.

2. Where to apply it

Decide whether the scope is project or global. Do not force an unsupported scope.

3. Merge or replace

If a matching file already exists, confirm whether to overwrite or merge it.

4. Whether to apply now

Review the dry-run summary and changed-file list before approving apply.

Explicit Approval Examples

Do not treat vague answers as approval. Be stricter on the first apply.

  • Good: yes, apply
  • Good: yes, overwrite and apply
  • Good: apply it at global scope
  • Not enough: sure, that one, go ahead, do it

Published Packs

Commit Convention Skill
skill id: commit-convention

Purpose: Standardize commit messages with a conventional format for clearer history and collaboration.

  1. Check the existing commit style in the current repository.
  2. Present the base format: <type>(<scope>): <summary>.
  3. Ask the user for apply scope (project / global).
  4. If rule files exist, ask for overwrite confirmation.
  5. Apply and provide two sample commit messages.
PR Clarity Template Skill
skill id: pr-clarity-template

Purpose: Keep PR descriptions concise and review-friendly with a consistent structure.

  1. Check whether a PR template already exists.
  2. Present sections: Summary, Changes, Scope, Validation, Related Issue.
  3. Ask the user for apply scope (project / global).
  4. Ask for merge/replace choice if a template already exists.
  5. Apply and provide one sample PR body.
